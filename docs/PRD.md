# Gravuresse vNext 产品需求文档（PRD）

> 状态：Baseline v0.5（现有架构优化 / Maintainable Creative Loop）
> 日期：2026-07-16
> 产品负责人：Daniel
> 文档目标：保留 Gravuresse 现有简洁实用的产品架构，在不推翻 Electron + React + Conversation/Asset 模型的前提下，系统优化创作流程、界面布局、组件体系、主题、语言和后续维护迭代能力。

---

## 0. v0.5 结论

老板给出的现有截图说明：Gravuresse 当前的产品架构是对的。

它不是需要被推翻的平台原型，而是一个已经形成清晰基本形态的本地 AI 视觉创作工具：

```text
顶部标题栏 / 设置入口
左侧模块栏
中间聊天与生成控制
右侧画布 / 素材区 / 素材详情
设置弹窗管理 Provider、主题、语言和功能开关
```

v0.5 的方向不是重构成新平台，而是把它从“简洁可用”打磨成：

```text
简洁但不简陋
强大但不复杂
能快速创作，也能长期维护迭代
```

从第一性原理看，AI 视觉创作工具的核心不是“工作流概念”，而是用户在每一步都不丢上下文、不丢结果、不丢继续创作的入口：

```text
表达意图 → 生成候选 → 选择结果 → 继续修改 / 转视频 → 整理素材 → 下次接着做
```

v0.5 的核心判断：

1. **保留当前双区工作形态**：聊天生成 + 画布素材是主界面，不新增复杂首页或平台壳。
2. **优化创作流程，而不是新增流程引擎**：用户说的“工作流”落到更顺的创作路径、状态反馈和继续创作入口，不变成节点系统。
3. **把主题、语言和工作区形态作为偏好系统补齐**：浅色默认，深色作为完整主题；中文/英文可切换；工作区形态在“画布模式”和“Pipeline 模式”之间二选一。
4. **把 Pipeline 和画布设为互斥形态**：使用 Pipeline 形式时取消网格和无限画布；不用 Pipeline 时就是网格 + 自由/无限画布形式。
5. **把组件体系正规化**：沉淀按钮、卡片、面板、状态、表单、弹窗等可复用组件规则。
6. **把维护性写进产品边界**：任何新增能力都先问能否挂在现有 Conversation / Message / Asset / Generation / ProviderConnection 上。
7. **继续拒绝大而全平台化**：不引入独立项目系统、节点流程、完整谱系、多 Agent 编排、云协作或大型数据库迁移作为 P0 前提。

一句话：**现有架构是地基；vNext 要做的是把地基加固、把动线理顺、把 UI 和组件体系整理成可长期生长的产品系统。**

## 1. 产品定义

### 1.1 一句话定位

**Gravuresse 是一个本地优先的 AI 视觉创作桌面应用：用户在当前创作记录里对话、生成图像/视频、管理素材，并把满意结果继续作为下一轮修改或视频生成的参考。**

它可以逐步变强，但当前阶段不把自己包装成复杂工作流平台。

### 1.2 北极星任务

> 让个人创作者在一个简洁的桌面界面里，从想法到图像/视频结果，再到继续修改和整理，都能顺畅闭环。

用户真正需要的是：

- 我能快速输入想法并生成图；
- 我能看懂当前用的是哪个模型和连接；
- 我能选中一张图继续改；
- 我能从一张图继续生成视频；
- 我能知道作品怎么来的；
- 我能切换浅色/深色和中文/英文；
- 我下次打开还能接着上次做；
- 应用以后继续加功能时不要越来越乱。

### 1.3 第一性原理

AI 视觉创作最小闭环由四件事构成：

| 第一性要素 | 用户语言 | 当前代码对应 | v0.5 处理 |
|---|---|---|---|
| 意图 | 我想做什么画面 | `Message.content` / composer | 输入更聚焦，参考摘要更清楚 |
| 结果 | 生成的图 / 视频 / 参考图 | `Asset` | 卡片、详情、筛选和继续动作更清楚 |
| 来源 | 这张图怎么来的 | `Asset.generation` / `MessageTask` | 详情页解释 prompt、模型、父素材、参考图 |
| 环境 | 用哪个连接和偏好 | `ProviderConnection` / `general` config | Provider、主题、语言、字号进入设置体系 |

任何新功能都必须回答：

1. 它是否缩短或稳定这条创作闭环？
2. 它是否可以挂在现有实体上？
3. 它是否会让默认界面变复杂？
4. 它是否会让后续维护更清楚，而不是更混乱？

## 2. 产品边界

### 2.1 P0 / P0.5 必须做

#### A. 创作流程优化

- 保持“聊天 + 画布/素材”的主路径；
- 输入栏显示当前会带入的参考素材、模型和关键参数；
- 生成中、完成、失败、需手动检查等状态必须在当前记录里可见；
- 选中素材后，“作为参考继续生成”“生成视频”“保存到本地”等动作必须直接可见；
- 失败时提供下一步，而不是只显示技术错误。

#### B. 界面布局优化

- 保留现有浅色双区架构作为默认；
- 左侧模块栏只放高频入口，不堆多层导航；
- 聊天区负责意图、生成控制和记录；
- 画布区负责素材查看、整理、筛选和自由摆放；
- 素材详情跟随选中素材出现，避免用户离开当前上下文；
- 设置继续使用弹窗/面板，不新建独立复杂后台。

#### C. 组件体系优化

沉淀以下可复用组件规则：

- `Button`：primary / secondary / ghost / danger / icon；
- `Pill`：状态、能力、参数、标签；
- `Panel`：玻璃/实色/浮层/设置区；
- `Toolbar`：画布工具、素材筛选、批量动作；
- `AssetCard`：图像、视频、参考素材、生成中、错误；
- `AssetDetail`：预览、生成说明、参考关系、继续动作；
- `TaskCard`：pending/running/completed/failed/unknown；
- `PreferenceRow`：语言、主题、字号、功能开关；
- `EmptyState` / `ErrorState`：空画布、无连接、生成失败。

#### D. 主题与语言

当前代码已经有基础：

- `general.theme = light | dark | system`；
- `general.language = zh | en`；
- `general.fontSize = small | medium | large`；
- `global.css` 有 light/dark/system token；
- `src/i18n.js` 有 zh/en 字典；
- `Settings.jsx` 已有通用设置入口。

v0.5 要做的是补齐体验与一致性：

- 默认主题：浅色；
- 深色主题：完整可用，不只是酷炫截图；
- 跟随系统：真实响应 OS 偏好；
- 中文/英文：主路径、设置、错误、状态都进入 i18n；
- 字号：不破坏布局；
- 新文案不得硬编码在组件里。

#### E. 维护迭代优化

- 不让 `App.jsx` 继续无限膨胀；
- 组件职责清楚，避免 UI、存储、Provider 调用混在同一层；
- 新字段先进入类型、sanitizer、导入导出测试；
- 视觉样式尽量回到 CSS token 和共享组件，不继续到处堆 inline style；
- 每个阶段都有可验收门禁，而不是“看起来完成”。

### 2.2 P0 明确不做

以下全部不进入 P0：

- 独立项目/分支/合并系统；
- 节点式流程编辑器；
- 把画布变成执行引擎；
- 完整版本树或大型来源图；
- 多会话上下文系统；
- 多 Agent 创作编排；
- 全新云同步、团队协作或权限系统；
- SQLite 或大型 schema 迁移；
- 未经用户要求新增默认 Provider。

这些不是永远不做，而是必须等当前简洁架构被真实使用证明不够用之后再做。

### 2.3 P1 可考虑

- 会话分组为真正项目；
- 更强的素材来源可视化；
- 更细的批量生成；
- 自定义创作模板；
- 更智能的提示词整理；
- 更完整的模型能力检测；
- 可折叠的高级参数面板；
- 用户自定义布局密度。

进入条件：**真实使用中出现明确痛点**，不是为了显得高级。

## 3. 目标用户与核心任务

### 3.1 目标用户

MVP 聚焦愿意配置 API、需要多轮图像/视频生成的个人创作者、设计学生、AI 视觉实验者和小型工作室主创。

他们的痛点是：

- 输出散落；
- 参数和参考图记不住；
- 想基于一张满意结果继续改很麻烦；
- 图转视频链路割裂；
- 任务失败后不知道下一步；
- 应用越加功能越难维护；
- 视觉主题和语言偏好不一致。

### 3.2 Jobs To Be Done

1. 当我有一个画面想法时，我想快速生成几张候选图。
2. 当我喜欢某张图的一部分时，我想基于它继续改，而不是重新开始。
3. 当我想做视频时，我想直接从一张图继续生成视频。
4. 当生成失败时，我想知道为什么失败以及能不能重试。
5. 当我隔天回来时，我想看到上次的聊天、素材和任务状态。
6. 当我长时间使用时，我想切换浅色/深色、中文/英文和合适字号。
7. 当应用继续升级时，我希望界面保持熟悉，不因为新功能变成陌生平台。

## 4. 用户界面原则

### 4.1 默认贴合当前产品

v0.5 默认界面基于当前真实截图：

```text
TitleBar
└─ WorkspaceShell
   ├─ ModuleSidebar
   └─ WorkspaceMain
      ├─ ChatPanel
      └─ CanvasPanel
         ├─ CanvasToolbar
         ├─ Grid / Free Canvas
         └─ AssetDetail（有选中素材时）
```

不要把默认 UI 改成大型三栏后台、营销首页或全新深色工作台。

### 4.2 深色是主题，不是新产品

深色界面可以很酷，但它应该是同一个产品的 `dark theme`，不是另一套信息架构。

要求：

- 浅色和深色使用同一布局；
- 同一组件只换 token，不换交互模型；
- 媒体内容保持主角；
- 暗色下边界、输入框、浮层和 disabled 状态要足够清楚。

### 4.3 语言是产品能力，不是翻译补丁

语言设置应满足：

- 默认中文；
- 可切换英文；
- 关键路径全覆盖：标题、设置、按钮、错误、状态、空状态；
- Provider 名称、模型 ID、taskId 不强行翻译；
- 新增文案必须先进入 i18n key。

### 4.4 简单来自低认知负担

默认界面只使用这些用户概念：

| 用户词 | 含义 |
|---|---|
| 创作记录 | 当前会话 |
| 作品 / 素材 | 图像、视频、参考图 |
| 生成说明 | prompt、模型、尺寸、参考图、父素材 |
| 作为参考继续生成 | 基于选中素材继续做图 |
| 生成视频 | 从选中图像发起视频任务 |
| 任务状态 | 运行中、完成、失败、需手动检查 |
| 连接 | 已配置的模型入口 |
| 主题 / 语言 | 用户偏好 |

复杂内部概念只允许出现在技术文档或高级折叠区，不进入默认主路径。

## 5. 功能需求

### 5.1 创作记录

- 用户可以创建、切换、删除会话；
- 每个会话保存 `messages` 和 `assets`；
- 会话标题可自动从首条消息生成，也可重命名；
- 当前会话切换时，聊天和画布资产同步切换；
- 导出单个会话或整个项目包时保留素材与生成说明；
- 会话列表可以折叠/弹出，避免长期占据主创作空间。

### 5.2 图像生成

- 支持文本生图；
- 支持参考图生图/改图能力，具体取决于当前 Provider；
- 输入栏展示当前比例、分辨率、风格、参考图摘要；
- 生成中显示占位和可理解状态；
- 成功后创建 Asset；
- `Asset.generation` 必须记录最小生成说明；
- 失败写回消息任务，不吞错误。

### 5.3 继续创作

- 资产卡片或详情提供“作为参考继续生成”；
- 图片资产提供“生成视频”；
- 新任务自动记录：`parentAssetId`、`sourceAssetIds`、`promptReferenceAssetIds`、prompt、参数、provider、model、taskId。

### 5.4 视频生成与恢复

- 支持从图像继续生成视频；
- 提交成功后立即保存远程 taskId；
- 队列显示 pending/running/completed/failed/unknown；
- 轮询成功后创建 video Asset；
- 应用重启后对 queued/running 任务执行恢复策略；
- 连接缺失或远程状态不明时显示“需手动检查”，不伪造成成功或失败。

### 5.5 连接和模型

- 使用当前 Provider connections；
- 模型选择只显示当前可用模型；
- 不新增未经用户配置的预设服务；
- Provider 限制在提交前尽量 precheck；
- 不支持时直接用人话报错；
- 密钥、token、cookie 不回显。

### 5.6 主题、语言和偏好

- 设置里提供：语言、主题、字号、工作区形态、参考图、视频工作区、自动保存历史；
- 修改语言/主题即时保存并即时生效；
- 主题值支持 `light | dark | system`；
- 语言值支持 `zh | en`；
- 字号改变不应造成主布局溢出；
- 工作区形态支持 `canvas | pipeline`，默认 `canvas`；
- `canvas` 表示网格 + 自由/无限画布；`pipeline` 表示流程节点视图；两者不得同时作为主工作区出现；
- 偏好设置不影响 Provider 凭据保存。

### 5.7 工作区形态：Pipeline 与画布互斥

设置中增加“工作区形态”开关：

| 模式 | 用户看到 | 必须隐藏/取消 | 适用场景 |
|---|---|---|---|
| `canvas` | 网格视图 + 自由/无限画布 + 素材详情 | Pipeline 主视图 | 日常生成、整理素材、选中继续创作 |
| `pipeline` | Pipeline 节点、执行顺序、节点输出和状态 | 网格视图、自由/无限画布入口 | 专家用户编排复杂生成链路 |

规则：

- 默认使用 `canvas`；
- 切到 `pipeline` 后，主工作区不再显示 Grid / Free Canvas / Infinite Canvas；
- 切回 `canvas` 后，主工作区恢复网格和自由/无限画布，不显示 Pipeline 节点主视图；
- 切换只改变工作区投影，不删除已有素材、消息或生成记录；
- 不承诺 Pipeline 与自由画布之间无损互转。

### 5.8 画布与素材

- 仅在 `workspaceMode='canvas'` 时保留网格视图和自由/无限画布；
- 网格视图用于快速浏览和选择素材；
- 自由/无限画布用于位置、缩放、查看和轻量标注；
- 支持 undo/redo 的范围限于画布资产操作；
- 支持素材筛选、导入、批量动作建议；
- 画布不是执行引擎，Pipeline 模式下不复用画布来伪装流程图。

## 6. 验收场景

### AC-01：文本生图闭环

用户新建/打开一个创作记录，输入 prompt，选择图像模型，生成成功后看到图片资产；资产详情显示 prompt、模型、比例和创建时间。

### AC-02：基于图片继续生成

用户选中一张图，点击“作为参考继续生成”，输入修改要求，新图片生成后记录父素材/参考素材关系。

### AC-03：图转视频

用户选中图片，点击“生成视频”，提交后出现队列任务；完成后视频进入资产区；失败时任务显示错误。

### AC-04：重启恢复

视频任务 queued/running 时关闭应用，再打开后：如果有足够信息恢复轮询，则继续轮询；如果信息不足或 Provider 不可用，则显示“状态未知/需要手动检查”；不伪造成失败或成功。

### AC-05：主题切换

用户在设置中从浅色切换深色，主界面、设置弹窗、输入框、画布工具和素材详情都切换到深色 token；切回浅色后布局不变。

### AC-06：语言切换

用户在设置中从中文切换英文，标题、按钮、设置项、状态、空状态和错误文案更新；模型 ID、Provider 技术字段保持原样。

### AC-07：工作区形态切换

用户在设置中选择“画布模式”时，主工作区显示网格/自由画布；选择“Pipeline 模式”时，主工作区只显示流程节点和节点输出，不再显示网格或无限画布入口；切换不删除素材、消息或生成说明。

### AC-08：组件一致性

同类按钮、状态标签、卡片、面板、错误态在不同页面表现一致；新增页面不得自造一套不兼容样式。

### AC-09：导入导出

导出的 `.gravuresse.json` 重新导入后，消息、资产、生成说明和可用媒体都保留；超大媒体按现有 fallback 规则处理。

## 7. 成功指标

### 7.1 产品指标

- 用户 3 分钟内能完成第一张图；
- 用户能从一张图继续做第二轮修改；
- 用户能从图片发起视频任务；
- 重启后会话和资产不丢；
- 生成失败时用户知道下一步；
- 用户能在 10 秒内找到主题和语言设置；
- 用户能在设置中明确切换“画布模式 / Pipeline 模式”，且不会同时看到两套主工作区；
- 新功能加入后主界面概念数量不明显增加。

### 7.2 工程指标

- `npm run test:core`、`npm run typecheck`、`npm run build` 作为基线门禁；
- import/export fixture 不回归；
- sanitizer 不丢 P0 必需字段；
- i18n key 覆盖主路径新增文案；
- 主题 token 覆盖新增组件；
- 新组件不重复实现同类基础样式；
- 不引入新数据库和大规模 schema 迁移。

## 8. Go / No-Go

### Go

可以进入开发的条件：

- 方案基于现有 `Conversation / Message / Asset / Generation / VideoQueueTask / ProviderConnection / general config`；
- 主题/语言复用当前设置系统；
- 新字段是少量 additive 字段；
- UI 基于当前 `ChatPanel / CanvasPanel / AssetDetail / Settings` 演进；
- 测试能覆盖保存、导入导出、生成元数据、任务恢复、主题和语言。

### No-Go

出现以下任一情况，立即回退：

- 需要新增独立项目/分支/节点流程等完整新模型；
- 需要把 Renderer 当前状态全部迁到主进程 Application Core；
- 需要新增通用 Agent 或任务编排系统；
- 需要 SQLite 或大型迁移才能开始；
- 默认界面让用户理解内部工程概念；
- 为了未来扩展牺牲当前简单可运行架构；
- 深色主题和英文语言变成另一套不可维护 UI。

## 9. 分阶段建议

### Phase 0：恢复工程基线

- 修复已知 `npm run test:core` 跨平台失败；
- 确认 `typecheck/build` 通过；
- 锁定当前真实架构和默认界面；
- 删除任何仍暗示新平台重构的 P0 承诺。

### Phase 1：偏好系统和设计 Token

- 验证 `theme/language/fontSize` 保存、生效和重启恢复；
- 补齐深色主题 token 覆盖；
- 补齐中文/英文主路径文案；
- 把新增文案统一进 `src/i18n.js`；
- 提炼基础状态和组件 token。

### Phase 2：创作流程打磨

- 输入栏展示当前参考、模型和参数；
- 素材卡片动作更直接；
- 素材详情解释生成说明；
- 失败和未知状态给下一步；
- 图转视频路径更短。

### Phase 3：任务恢复和生成说明

- 补齐 `Asset.generation`；
- 补齐 `MessageTask` 可恢复字段；
- sanitizer 和 import/export 测试跟上；
- 视频任务重启后恢复或明确 unknown/not-resumable。

### Phase 4：组件与维护性整理

- 从高重复区域开始抽共享组件；
- 降低 `App.jsx` / `CanvasPanel.jsx` / `ChatPanel.jsx` 单文件复杂度；
- 保持行为不变，小步提交；
- 每次抽组件都用截图和测试验证。

## 10. 变更规则

v0.5 之后，任何新增需求必须标注：

1. 它优化哪一步创作闭环；
2. 它是否可复用现有实体；
3. 它是否增加默认界面概念；
4. 它是否已有 i18n 和 theme token；
5. 它是否会让维护边界更清楚。

默认判断：**能优化当前架构，就不重做架构；能用现有组件，就不自造新样式；能折叠到设置，就不污染主路径。**
