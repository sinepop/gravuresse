# Gravuresse vNext 产品需求文档（PRD）

> 状态：Baseline v0.4（现有架构优先 / Simple Architecture）
> 日期：2026-07-16
> 产品负责人：Daniel
> 文档目标：把 v0.3 过度平台化的设想收敛回当前 Gravuresse 的真实架构：**不重做一套“项目/工作流/Agent 平台”，只在现有会话、画布、素材和连接体系上补最小创作闭环。**

---

## 0. v0.4 结论

v0.3 的问题不是 UI 不够简单，而是 **PRD/SPEC 把一个已经能工作的本地桌面应用，推向了新平台重构**。

从第一性原理看，当前阶段 Gravuresse 最重要的不是建立完整创意工作台理论，而是让用户在现有产品里更顺地完成：

```text
输入想法 → 生成图像 → 保存/整理结果 → 选中结果继续修改或转视频 → 下次打开还能接着做
```

因此 v0.4 的核心判断：

1. **当前会话就是当前 MVP 的项目单元**：暂不新增独立项目、分支或合并模型。
2. **消息就是创作记录**：用户与助手的对话、任务状态、错误和结果说明都留在消息里。
3. **素材就是创作结果**：图像/视频、参考图、父子关系和生成元数据都围绕素材展开。
4. **画布只是整理和查看素材的界面**：保留网格/自由视图，不升级为工作流编辑器。
5. **生成元数据就是最小来源记录**：记录 prompt、模型、比例、父素材、参考素材和远程 taskId；不另建完整谱系/版本系统。
6. **连接沿用现有体系**：不另造能力协商系统，只做清晰的可用性提示和错误反馈。
7. **异步视频任务只做最小恢复**：让 queued/running 任务重启后能继续轮询或明确失败；不做通用任务引擎。

一句话：**先把现有架构打磨成“简单、可靠、可继续创作”，不要把它改造成未来幻想中的大型创意平台。**

---

## 1. 产品定义

### 1.1 一句话定位

**Gravuresse 是一个本地优先的 AI 视觉创作桌面应用：用户在一个创作记录里对话、生成图像/视频、管理素材，并把满意结果继续作为下一轮修改或视频生成的参考。**

它不是首版工作流引擎，也不是节点工具，也不是多 Agent 创意平台。

### 1.2 当前版本的北极星任务

> 让一次 AI 视觉创作在一个本地会话里闭环：从想法到结果，再从结果继续修改、整理或生成视频。

用户真正需要的不是理解工作流，而是：

- 我说一句想法，能出图；
- 我选中一张图，能继续改；
- 我选中一张图，能转视频；
- 我能看到这张图大概怎么来的；
- 我下次打开时，素材和记录还在。

### 1.3 v0.4 第一性原理

AI 视觉创作的最小闭环只有四个实体：

| 实体 | 用户理解 | 当前代码里的对应 |
|---|---|---|
| 创作记录 | 这一轮聊天和生成过程 | `Conversation.messages` |
| 作品/素材 | 生成图、视频、参考图 | `Conversation.assets` / `Asset` |
| 生成说明 | prompt、模型、尺寸、父素材 | `Asset.generation` / `MessageTask` |
| 连接 | 用哪个模型生成 | Provider connections / model selector |

任何新概念都必须回答：**没有它，用户是否无法完成最小闭环？**

如果答案是否，它就不能进入 P0。

## 2. 产品边界

### 2.1 P0 必须做

P0 只补强当前架构能承载的事情：

1. **创作记录稳定**
   - 会话列表、当前会话、标题、消息和资产保存稳定；
   - 删除、切换、导入导出不丢素材；
   - 当前会话就是默认工作空间。

2. **图像生成闭环清楚**
   - 用户输入 prompt / 选择风格、比例、尺寸；
   - 生成成功后自动进入资产区；
   - 失败时错误可理解，可重试；
   - 资产详情能看到生成说明。

3. **基于已有结果继续创作**
   - 选中资产后可“作为参考继续生成”；
   - 选中图像后可“生成视频”；
   - 新结果记录父素材和参考素材 ID；
   - 不要求用户理解分支、谱系或工作流。

4. **视频异步任务最小可靠**
   - queued/running 状态可见；
   - 远程 `taskId` 一旦拿到就保存；
   - 应用重启后能恢复轮询，或者明确显示“状态未知，需要手动重试/检查”；
   - 重试不覆盖原始失败记录。

5. **连接设置可用**
   - 沿用现有连接、模型选择和验证；
   - 用户只需要知道当前连接能不能生成图/视频/对话；
   - 不新增默认 Provider，不擅自替用户预设服务。

6. **资产管理够用**
   - 网格查看；
   - 自由视图整理；
   - 详情查看；
   - 删除、保存到磁盘、导出项目。

### 2.2 P0 明确不做

以下全部移出 P0：

- 独立项目/分支/合并系统；
- 工作流图、步骤引擎和任意节点编排；
- 把画布升级成独立业务真相；
- 完整谱系、版本树和复现评级系统；
- 多会话上下文快照系统；
- 多 Agent 或通用任务编排；
- 通用模型能力协商框架；
- SQLite 或全新本地数据库；
- 新的云同步、团队协作、权限系统。

这些不是永远不做，而是 **必须等当前简单架构证明不够用以后再做**。

### 2.3 P1 才考虑

P1 可以考虑，但不得提前污染 P0：

- 会话分组为真正项目；
- 更强的来源可视化；
- 更明确的多方向探索；
- 批量生成；
- 更细的模型能力检测；
- 自定义工作流模板；
- 单助手辅助整理 prompt 或改图建议。

P1 的进入条件：P0 用户真实使用中出现明确痛点，而不是我们预先想象系统应该高级。

## 3. 目标用户与核心任务

### 3.1 目标用户

MVP 聚焦已经愿意配置 API、需要多轮图像/视频生成的个人创作者、设计学生、AI 视觉实验者和小型工作室主创。

他们的痛点是：

- 输出散落；
- 生成参数和参考图记不住；
- 想基于一张满意结果继续改很麻烦；
- 图转视频链路割裂；
- 应用重启后任务状态不可靠。

他们不需要一开始就面对工作流、谱系图、能力协商、Agent 编排。

### 3.2 Jobs To Be Done

1. 当我有一个画面想法时，我想快速生成几张候选图。
2. 当我喜欢某张图的一部分时，我想基于它继续改，而不是重新开始。
3. 当我想做视频时，我想直接从一张图继续生成视频。
4. 当生成失败时，我想知道为什么失败以及能不能重试。
5. 当我隔天回来时，我想看到上次的聊天、素材和任务状态。
6. 当我导出项目时，我想带走会话、素材和生成说明。

## 4. 用户界面原则

### 4.1 保留当前基本形态

v0.4 不要求推翻当前 UI。默认仍然是：

```text
左侧/主区：聊天与生成控制
右侧/主区：画布/资产结果
顶部/设置：连接、模型、应用设置
```

v0.3 的 UI 原型可以作为视觉参考，但不是架构蓝图。尤其不能为了“步骤视图”去新增 Workflow 系统。

### 4.2 简单来自少概念，不是多包装

默认界面只允许出现这些用户概念：

| 用户词 | 含义 |
|---|---|
| 创作记录 | 当前会话 |
| 作品 / 素材 | Asset |
| 生成说明 | prompt、模型、尺寸、参考图 |
| 参考这张图 | 使用 `sourceAssetIds` / `parentAssetId` |
| 继续生成视频 | image-to-video task |
| 连接 | 已配置的模型入口 |

默认界面不出现复杂架构词：工作流、配方、谱系、能力协商、任务尝试、上下文快照、Agent 运行等。

### 4.3 资产详情是透明度入口

不做复杂谱系图，先把资产详情做好：

- prompt；
- negative prompt；
- provider / model；
- ratio / resolution / duration；
- 父素材；
- 参考素材；
- 远程 taskId；
- 创建时间；
- 错误信息（如有）。

这已经覆盖 P0 需要的“知道它怎么来的”。

## 5. 功能需求

### 5.1 创作记录

- 用户可以创建、切换、删除会话；
- 每个会话保存 `messages` 和 `assets`；
- 会话标题可自动从首条消息生成；
- 当前会话切换时，聊天和画布资产同步切换；
- 导出单个会话或整个项目包时保留素材与生成说明。

### 5.2 图像生成

- 支持文本生图；
- 支持参考图生图/改图能力，具体取决于当前 Provider；
- 生成中显示占位；
- 成功后创建 Asset；
- Asset.generation 必须记录最小生成说明；
- 失败写回消息任务，不吞错误。

### 5.3 视频生成

- 支持从图像继续生成视频；
- 提交成功后立即保存远程 taskId；
- 队列显示 pending/running/completed/failed；
- 轮询成功后创建 video Asset；
- 应用重启后对 queued/running 任务执行恢复策略。

### 5.4 继续创作

- 资产卡片或详情提供“作为参考继续生成”；
- 图片资产提供“生成视频”；
- 新任务自动记录：
  - `parentAssetId`；
  - `sourceAssetIds`；
  - `promptReferenceAssetIds`；
  - prompt 和参数；
  - provider / model；
  - taskId（如果有）。

### 5.5 连接和模型

- 使用当前 Provider connections；
- 模型选择只显示当前可用模型；
- 不新增未经用户配置的预设服务；
- Provider 限制（prompt 长度、时长、比例、参考图要求）在提交前尽量 precheck；
- 不支持时直接用人话报错。

### 5.6 画布与资产

- 保留网格视图和自由视图；
- 自由视图只管理资产位置、缩放、查看，不承担工作流语义；
- 支持 undo/redo 的范围限于画布资产操作；
- 不承诺自由画布与步骤视图互转，因为 P0 没有步骤视图领域模型。

## 6. 验收场景

### AC-01：文本生图闭环

用户新建/打开一个创作记录，输入 prompt，选择图像模型，生成成功后看到图片资产；资产详情显示 prompt、模型、比例和创建时间。

### AC-02：基于图片继续生成

用户选中一张图，点击“作为参考继续生成”，输入修改要求，新图片生成后记录父素材/参考素材关系。

### AC-03：图转视频

用户选中图片，点击“生成视频”，提交后出现队列任务；完成后视频进入资产区；失败时任务显示错误。

### AC-04：重启恢复

视频任务 queued/running 时关闭应用，再打开后：

- 如果有足够信息恢复轮询，则继续轮询；
- 如果信息不足或 Provider 不可用，则显示“状态未知/需要手动检查”，不伪造成失败或成功。

### AC-05：导入导出

导出的 `.gravuresse.json` 重新导入后，消息、资产、生成说明和可用媒体都保留；超大媒体按现有 fallback 规则处理。

### AC-06：无复杂概念泄漏

默认界面和 P0 文案不出现复杂架构词：工作流、配方、谱系、能力协商、任务尝试、上下文快照、Agent 运行等。

---

## 7. 成功指标

### 7.1 产品指标

- 用户 3 分钟内能完成第一张图；
- 用户能从一张图继续做第二轮修改；
- 用户能从图片发起视频任务；
- 重启后会话和资产不丢；
- 生成失败时用户知道下一步该怎么做。

### 7.2 工程指标

- `npm run test:core`、`npm run typecheck`、`npm run build` 作为基线门禁；
- import/export fixture 不回归；
- sanitizer 不丢 P0 必需字段；
- 异步视频任务恢复有自动化测试；
- 不引入新数据库和大规模 schema 迁移。

---

## 8. Go / No-Go

### Go

可以进入开发的条件：

- 方案只改现有 `Conversation / Message / Asset / Generation / VideoQueueTask / ProviderConnection`；
- 新字段是少量 additive 字段；
- 不需要全量迁移老项目；
- UI 基于当前 ChatPanel / CanvasPanel / AssetDetail 演进；
- 测试能覆盖保存、导入导出、生成元数据和任务恢复。

### No-Go

出现以下任一情况，立即回退：

- 需要新增独立项目/分支/工作流/谱系等完整新模型；
- 需要把 Renderer 当前状态全部迁到主进程 Application Core；
- 需要新增通用 Agent 或任务编排系统；
- 需要 SQLite 或大型迁移才能开始；
- 默认界面让用户理解工作流、配方、谱系等内部词；
- 为了未来扩展牺牲当前简单可运行架构。

---

## 9. 变更规则

v0.4 之后，任何新增需求必须标注：

1. 它解决哪个 P0 用户闭环问题；
2. 是否可以用现有 Conversation/Asset/Task 结构表达；
3. 是否需要新增顶层领域模型；
4. 如果需要新增模型，为什么不能放到 P1。

默认判断：**能用现有架构表达，就不新增架构。**
