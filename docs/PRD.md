## 1. 产品定义

### 1.1 一句话定位

**Gravuresse 是一个面向图像与视频创作的本地 AI 创意工作台：用户通过自然语言表达脑海中的想法，由可控的 Agent 协作系统将意图整理为生成方案，并在 Pipeline 或无限画布中持续生成、比较、修改和复现。**

### 1.2 北极星任务

用户真正需要的不是“生成一张图”，而是：

> 把尚不精确的脑内意象，逐步变成可观察、可比较、可修改、可复现的图像或视频，同时保留创作过程中的语义、上下文和选择依据。

### 1.3 产品参考的正确用法

Gravuresse 吸收 Lovart 一类 AI 创意工作区的“对话驱动设计”和生图生视频工具的“媒体生成能力”，但不复制其界面或工作流。差异点应落在：

1. 用户可自由连接自己的 API；
2. Agent 协作过程可控、可查看、可停止；
3. 同一创意可以在 Pipeline 与无限画布之间转换；
4. 每次生成与修改都保留谱系和参数快照；
5. 多会话既隔离又能显式共享项目上下文。

## 2. 第一性原理与产品边界

### 2.1 创作误差从哪里产生

从脑内想法到最终画面，误差主要来自四层翻译：

```text
模糊意图
→ 可描述的创意需求
→ 模型可执行的提示词/参数
→ 模型实际输出
→ 用户对结果的再次判断
```

Gravuresse 的价值不是消除随机性，而是缩短每轮误差，并把修改过程保存为可重复的决策链。

### 2.2 三项不可虚假承诺

1. **不承诺“完美保存无限上下文”**：不同模型有不同上下文窗口。产品提供项目记忆、会话分支、上下文快照、固定素材和显式引用，保证上下文可管理、可追溯，而非无限原文注入。
2. **不承诺跨模型像素级复现**：部分供应商不支持 seed，模型版本也可能漂移。产品保存可获得的模型版本、参数、参考素材、提示词、蒙版和响应证据，并给出复现可信度。
3. **不承诺 Agent 越多结果越准**：Agent 只在任务可以分工时启用；最终由用户确认创意简报和候选结果，避免自动循环消耗成本。

### 2.3 产品原则

1. **意图先于提示词**：先让用户确认创意简报，再适配不同模型的 prompt。
2. **结果必须可追溯**：任何输出都能回到来源素材、会话、模型、参数和父版本。
3. **控制逐层展开**：默认黑盒简单操作，需要时切换灰盒/白盒，而不是把参数一次性压给用户。
4. **项目共享，会话隔离**：项目记忆可复用，会话历史默认不互相污染。
5. **能力驱动而非 Provider 驱动**：UI 展示“可做什么”，底层再选择已验证连接。
6. **用户始终拥有决策权**：高成本生成、批量任务、自动迭代均需可见预算和停止点。
7. **本地优先**：项目、上下文、参数和密钥由本地桌面应用管理；不为了方便擅自上传敏感信息。

## 3. 目标用户与核心任务

### 3.1 主要用户

#### A. 有明确画面但不懂模型参数的创作者
- 能描述感觉、故事和参考作品，但不会写稳定 prompt；
- 需要快速把模糊想法变成可比较的画面；
- 更适合 Pipeline 和黑盒模式。

#### B. 需要高控制度的设计师、导演和 AI 创作者
- 需要锁定角色、构图、镜头、风格或品牌元素；
- 会比较模型、参数、参考权重和迭代差异；
- 更适合无限画布和白盒模式。

### 3.2 Jobs To Be Done

1. 当我脑中有一个不完整的视觉想法时，帮助我把它澄清为可执行的视觉方案。
2. 当第一次生成不准确时，让我只修改需要变化的部分，不破坏满意部分。
3. 当我在多个模型和供应商之间选择时，告诉我当前连接能做什么，而不是让我记 API 差异。
4. 当我同时探索多个方向时，让每个会话保持完整脉络，并能把好结果带到另一个分支。
5. 当我从图片继续做视频时，保留角色、场景、镜头和风格连续性。
6. 当我再次打开项目或交给其他人时，能够解释结果如何产生并尽可能复现。

## 4. 核心概念

### 4.1 Project（项目）

最高层创作容器，包含：
- 项目简报与目标；
- 共享记忆与固定素材；
- 多个会话；
- Pipeline/画布视图；
- 生成任务与输出谱系；
- Provider 连接引用（不在项目文件中保存明文密钥）。

### 4.2 Conversation（会话）

项目内的独立探索分支。每个会话拥有自己的原始消息、上下文快照、任务和素材引用。会话之间默认隔离，可显式：
- Fork：从当前上下文创建分支；
- Pin：把结论或素材提升到项目记忆；
- Reference：引用另一会话的指定快照；
- Merge brief：只合并确认后的创意简报，不盲目拼接全部聊天记录。

### 4.3 Creative Brief（创意简报）

用户意图的结构化中间层，不等同于最终 prompt。最小字段：
- 目标与用途；
- 主体；
- 场景与环境；
- 构图/镜头；
- 动作与时间；
- 风格与材质；
- 光线与色彩；
- 必须保留；
- 必须避免；
- 参考素材；
- 不确定项与待确认问题。

### 4.4 Generation Recipe（生成配方）

一次可执行生成任务的冻结快照：
- brief 版本；
- prompt / negative prompt；
- 模型、连接和能力版本；
- 比例、分辨率、时长、seed（若支持）；
- 参考素材及其 hash/权重；
- 蒙版/区域约束；
- Provider 特有参数；
- 父素材与修改说明；
- 创建者（用户/Agent）和时间。

### 4.5 Lineage（输出谱系）

输出与其父版本、来源素材、配方、会话和任务之间的有向关系。谱系必须支持：
- 查看父子版本；
- 对比 prompt/参数差异；
- 回到任一版本继续分支；
- 标记“最终”“候选”“废弃”；
- 导出配方和素材清单。

## 6. 核心功能需求

优先级：P0 = vNext 基础闭环；P1 = 稳定后扩展；P2 = 未来探索。

### 6.1 项目与双视图

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| PROJ-01 | 新建项目时选择 Pipeline 或无限画布 | P0 | 两种入口创建同一 Project，不是两种文件格式 |
| PROJ-02 | 项目内可切换 Pipeline/画布 | P0 | 数据不丢失；Pipeline 节点映射到画布节点 |
| PROJ-03 | Pipeline 可“一键展开为画布” | P0 | 保留节点状态、素材、任务和连接关系 |
| PROJ-04 | 画布修改若超出 Pipeline 表达能力，切回时提示“只读映射/需要整理” | P0 | 不静默丢节点或参数 |
| PROJ-05 | 项目自动保存、版本迁移、导入导出 | P0 | 崩溃恢复后保留最近一次完整写入 |

### 6.2 Pipeline 工作流

默认模板只包含用户当前确认的最小创作链：

```text
想法输入
→ 创意简报
→ 图像探索
→ 选定/修改
→ 视频生成（可选）
→ 导出
```

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| PIPE-01 | 每一步显示输入、输出、状态和执行者 | P0 | 用户知道当前卡在哪里 |
| PIPE-02 | 支持跳过可选步骤和从任一步重跑 | P0 | 不强制完整线性流程 |
| PIPE-03 | 每一步可选择自动路由或指定连接/模型 | P0 | 自动模式能解释选择依据 |
| PIPE-04 | Pipeline 节点支持多候选输出和人工选定 | P0 | 未选定前不自动覆盖下游 |
| PIPE-05 | 后续允许保存自定义 Pipeline 模板 | P1 | 本版先冻结系统默认模板 |

### 6.3 对话、上下文与多窗口

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| CTX-01 | 同一项目支持多个会话并行 | P0 | 切换会话不串消息、不串素材状态 |
| CTX-02 | 会话可 Fork，保留来源快照 | P0 | 新分支能回溯父会话和分叉点 |
| CTX-03 | 项目记忆与会话历史分层 | P0 | 只有 Pin 内容自动进入项目共享层 |
| CTX-04 | 发送前展示本轮上下文来源 | P0 | 用户能看到本轮引用了哪些素材、简报、记忆、历史摘要 |
| CTX-05 | 超出上下文预算时生成版本化摘要 | P0 | 原文不删除；摘要可检查、回滚、重建 |
| CTX-06 | 支持独立窗口打开会话 | P1 | 同项目多窗口同步冲突有明确规则 |
| CTX-07 | 显示上下文预算和裁剪原因 | P1 | 不以“完整上下文”模糊承诺 |

### 6.4 Agent 协作

默认不是角色扮演群聊，而是任务图：

```text
意图解析
→ Prompt/镜头方案并行
→ 连续性与约束检查
→ 模型能力适配
→ 候选评分与差异说明
→ 用户确认
```

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| AGT-01 | 单 Agent 与 Agent 协作两种执行模式 | P0 | 简单任务默认单 Agent，不强制集群 |
| AGT-02 | 协作运行前显示步骤、模型、预计调用次数 | P0 | 用户确认后执行，可随时停止 |
| AGT-03 | 每个 Agent 输出结构化结果，不把内部闲聊塞进主会话 | P0 | 主会话只保留结论、差异和证据 |
| AGT-04 | Prompt 方案支持并行候选和批判性审查 | P0 | 至少说明每个方案改变了什么 |
| AGT-05 | 连续性检查覆盖主体、场景、构图、光线、材质和禁用项 | P0 | 输出可直接更新 Brief 或生成配方 |
| AGT-06 | 支持保存自定义 Agent 方案 | P1 | 权限、成本上限和工具范围可配置 |
| AGT-07 | 自动多轮优化有次数/成本/时间上限 | P1 | 绝不无限自主循环 |

### 6.5 黑盒、灰盒、白盒修改

#### 黑盒模式（默认）
用户只描述“哪里不对、想怎么改”。系统自动选择能力、保持稳定项并生成配方；执行前以简短摘要告知将改变/锁定什么。

#### 灰盒模式
用户控制高价值语义项：主体、构图、镜头、光线、风格、变化强度、参考图权重和稳定锁；隐藏 Provider 细节。

#### 白盒模式
展示完整 prompt diff、negative prompt、模型/版本、seed、尺寸、采样/Provider 参数、参考图和蒙版等。只显示当前连接实际支持的参数。

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| EDIT-01 | 三种透明度模式可切换 | P0 | 切换不丢配方；白盒信息来自真实能力 |
| EDIT-02 | “稳定锁”可锁定主体、构图、风格、场景、镜头、色彩 | P0 | 锁定项进入 Brief 和 Recipe，不只做 UI 标签 |
| EDIT-03 | 修改采用非破坏分支 | P0 | 原素材和原配方保持不变 |
| EDIT-04 | 显示 Prompt/参数差异 | P0 | 用户能判断为什么发生变化 |
| EDIT-05 | 输出复现可信度 | P0 | 高/中/低并解释：seed、版本、参考素材、Provider 漂移等 |
| EDIT-06 | 区域修改、蒙版与局部重绘 | P1 | 仅在模型能力支持时出现 |
| EDIT-07 | 角色/风格一致性包 | P1 | 可将多素材与约束保存为可复用资产 |

### 6.6 图像与视频生成

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| GEN-01 | 文生图、图生图/编辑 | P0 | 根据连接能力显式启用 |
| GEN-02 | 文生视频、图生视频 | P0 | 异步任务进入统一任务队列 |

### 6.7 API 与连接

| ID | 需求 | 优先级 | 验收摘要 |

## 7. 关键用户流程

### 7.1 从想法到第一组图

```text
新建项目 → 选择 Pipeline
→ 用自然语言描述想法 + 添加参考
→ Agent 提问缺失信息
→ 用户确认 Creative Brief
→ 系统生成 2–4 个 Prompt 方案与差异
→ 用户选方案/自动适配连接
→ 冻结 Recipe 并生成候选
→ 用户标记满意项和问题
```

### 7.2 稳定修改

```text
选择候选 → 说“保留人物和构图，把时间改成清晨”
→ 黑盒模式自动识别：锁主体/构图，改光线/色彩
→ 用户查看变更摘要
→ 创建非破坏分支 → 生成
→ 在谱系中对比父子结果与配方差异
```

### 7.3 从图到视频

```text
选定图片 → 创建视频任务
→ Motion Agent 补齐镜头、动作、节奏、时长
→ 用户确认视频 Brief
→ 选择支持图生视频的连接
→ 冻结 Recipe → 排队 → 轮询 → 生成视频节点
```

### 7.4 并行探索与合并

```text
会话A探索写实方向 → Fork 会话B探索插画方向
→ 两边历史互不污染
→ 将A的角色设定和B的色彩方案 Pin到项目
→ 新会话显式引用两份项目记忆
```

### 7.5 Pipeline 展开为无限画布

```text
Pipeline右上角“展开为画布”
→ 每个步骤变成节点组
→ 候选输出保留在对应步骤下
→ 当前选中路径高亮
→ 用户可自由添加分支或新的生成节点
```

4. 单 Agent + 有限 Agent 协作；
5. 多会话隔离、Fork、Pin、上下文来源面板；
6. 黑/灰/白盒编辑；
7. 稳定锁、Recipe 快照、谱系；

- 自定义 Agent 方案；

- 自动发布到第三方平台；
- 插件市场；
- 无限制自主 Agent 循环。

## 9. 成功指标

### 9.1 核心指标

- **意图收敛轮次**：从初始描述到用户确认 Brief 所需轮数；
- **满意结果命中率**：一次候选批次中至少出现一个被用户收藏/选定结果的比例；
- **局部修改保留率**：用户声明的锁定项在下一轮仍被主观判定为保留的比例；
- **谱系可追溯率**：输出能够还原有效 Recipe 和父素材的比例；
- **上下文误引用率**：模型错误引用其他会话私有信息的比例；
- **任务成功率**：经过能力预检后实际完成生成的任务比例。

### 9.2 体验指标

- 新用户能在不打开白盒参数的情况下完成“想法→图片→修改”；
- 专业用户能查看并修改完整配方；
- 用户能在 10 秒内理解当前任务由谁执行、使用什么连接、预计发生什么；
- 任何高成本或批量调用均可在执行前确认、执行中停止。

## 10. 验收场景

### AC-01：黑盒生图闭环

给定一个新用户输入“雨夜里的中式未来城市，一位撑红伞的人走过老街”，系统能生成结构化 Brief，用户确认后得到候选图；用户无需填写 seed/采样参数。

### AC-02：白盒复现

选择任一输出，检查器能显示可获得的连接、模型版本、prompt、negative prompt、比例、分辨率、seed、参考素材、父版本与 Provider 参数；不支持的字段明确标记为不可用，而不是伪造默认值。

### AC-03：稳定修改

用户要求“只把雨夜改成清晨，人物、红伞和构图不变”，系统把锁定项和变化项写入新 Recipe，生成结果作为原图子分支保留。

### AC-04：多会话隔离

项目有 A/B 两个会话；B 不引用 A 时，不应自动注入 A 的私有消息。A 的内容只有 Pin 到项目或被显式 Reference 后才可用于 B。

### AC-05：Pipeline/画布转换

含 Brief、两组图像候选和一项视频任务的 Pipeline 能展开为画布，节点、状态、候选和谱系完整；切回 Pipeline 时，无法表达的自由节点被保留并给出提示。

### AC-06：连接安全

导出项目、应用日志、渲染进程状态和错误信息中不得出现明文 API Key、token、Cookie 或认证头。

## 11. 主要风险与对策

| 风险 | 后果 | 对策 |
|---|---|---|
| 目标过大，一次重写所有功能 | 长期无法发布 | 复用现有 Provider、会话、画布与任务队列；按领域模型增量迁移 |
| Agent 协作成本高但提升有限 | 用户付费增加、等待变长 | 默认单 Agent；复杂任务才出现协作建议；显示调用预算 |
| 不同 Provider 参数差异巨大 | 白盒界面失控 | 通用 Recipe + capability schema + namespaced extraParams |
| 上下文摘要漂移 | 创意设定被错误改写 | 摘要版本化、保留原文、用户可 Pin/纠正/回滚 |
| “稳定锁”被用户理解为绝对保证 | 信任受损 | 显示复现可信度和模型能力边界，不使用“100%一致”措辞 |
| Pipeline 与自由画布双向同步复杂 | 数据丢失或状态冲突 | 共享图模型；Pipeline 是受限投影视图，不维护第二份业务数据 |
| API 密钥和自定义端点引入安全风险 | 凭据泄漏/SSRF | 主进程持有密钥、端点验证、IPC allowlist、日志脱敏、导出过滤 |
| 多窗口并发写入 | 会话覆盖 | P1 再开放；引入 revision/optimistic concurrency 后实施 |

## 12. 变更规则

后续想法按以下格式进入 PRD，不直接塞进开发任务：

```text
新想法：
解决谁的什么问题：
进入哪个核心流程：
是否改变数据模型：
优先级建议：P0 / P1 / P2
不做会造成什么：
```

只有经过产品范围审查的需求，才能进入 `SPEC.md` 和开发任务。
