# Gravuresse vNext 产品需求文档（PRD）

> 状态：Baseline v0.3（Simple First 对抗式审查后收敛）
> 日期：2026-07-16
> 产品负责人：Daniel
> 文档目标：冻结“默认简单、高级可展开”的 P0 产品边界，作为技术 SPEC、UI 原型和后续开发的共同依据。后续新想法以变更记录追加，不在本版提前扩 scope。

---

## 1. 产品定义

### 1.1 一句话定位

**Gravuresse 是一个面向高频 AI 视觉创作者的本地优先创意工作台：用户默认只需要“描述想法、生成候选、选择结果、说出修改”，系统在后台把意图整理为可检查、可追溯、可恢复的创作流程。Pipeline、自由画布、生成设置、来源记录和模型能力都存在，但必须按需展开，不得在首屏压给用户。**

### 1.2 北极星任务

用户真正需要的不是“生成一张图”，而是：

> 把尚不精确的脑内意象，逐步变成可观察、可比较、可修改、可复现的图像或视频，同时保留创作过程中的语义、上下文和选择依据。

### 1.3 产品参考的正确用法

Gravuresse 吸收 Lovart 一类 AI 创意工作区的“对话驱动设计”和生图生视频工具的“媒体生成能力”，但不复制其界面或工作流。差异点应落在：

1. 用户可自由连接自己的 API，但连接设置必须像向导，而不是开发者控制台；
2. 助手辅助执行过程可控、可查看、可停止，但默认只显示“正在做什么”和下一步；
3. 同一创意默认走步骤式创作路径，必要时可展开为自由画布；
4. 每次生成与修改都保留来源记录和参数快照，但默认不把 Recipe/Lineage 等内部术语暴露给用户；
5. 多会话既隔离又能显式共享项目上下文，但主路径只显示“本轮参考了什么”。

### 1.4 Simple First 决策

对抗式审查后的核心修正：**底层可以复杂，默认界面必须让用户觉得自己只是在描述、选择和修改。**

P0 体验目标：

```text
30 秒内知道怎么开始
3 分钟内完成一次：描述 → 生成 → 选择 → 修改
```

因此：

- 新建项目不再让用户先选择 Pipeline / 无限画布；默认只有“开始创作”；
- Pipeline 是内部与高级文档术语，用户主界面称为“步骤视图”或“创作步骤”；
- 自由画布不是首屏同级模式，而是“高级：展开为自由画布”；
- 黑盒/灰盒/白盒不作为首屏三标签出现，默认只显示“简单修改”，高级设置再展开稳定锁、参数和来源；
- Recipe、Lineage、Capability、Provider、ContextSnapshot、Attempt 等术语不得出现在默认用户路径；
- Agent 默认称为“助手”，只展示阶段、结论、证据和停止按钮，不展示内部任务图或角色群聊。

## 2. 第一性原理与产品边界

### 2.1 创作误差从哪里产生

从脑内想法到最终画面，误差主要来自四层翻译：

```text
模糊意图
→ 可描述的创意需求
→ 模型可执行的提示词/参数
→ 模型实际输出
→ 用户对结果的再次判断
```

Gravuresse 的价值不是消除随机性，而是缩短每轮误差，并把修改过程保存为可重复的决策链。

### 2.2 三项不可虚假承诺

1. **不承诺“完美保存无限上下文”**：不同模型有不同上下文窗口。产品提供项目记忆、会话分支、上下文快照、固定素材和显式引用，保证上下文可管理、可追溯，而非无限原文注入。
2. **不承诺跨模型像素级复现**：部分供应商不支持 seed，模型版本也可能漂移。产品保存可获得的模型版本、参数、参考素材、提示词、蒙版和响应证据，并给出复现可信度。
3. **不承诺 Agent 越多结果越准**：Agent 只在任务可以分工时启用；最终由用户确认创意简报和候选结果，避免自动循环消耗成本。

### 2.3 产品原则

1. **Simple First**：默认路径只呈现描述、生成、选择、修改四件事；高级概念按需展开。
2. **意图先于提示词**：先让用户确认创作说明，再适配不同模型的 prompt。
3. **结果必须可追溯，但默认不打扰**：任何输出都能回到来源素材、会话、模型、参数和父版本；主界面只用“来源记录/版本”表达。
4. **控制逐层展开**：默认简单修改，需要时打开高级设置，而不是把参数一次性压给用户。
5. **项目共享，会话隔离**：项目记忆可复用，会话历史默认不互相污染。
6. **能力驱动而非供应商驱动**：UI 展示“能不能生成图/视频/对话”，底层再选择已验证连接。
7. **用户始终拥有决策权**：高成本生成、批量任务、自动迭代均需可见预算和停止点。
8. **本地优先**：项目、上下文、参数和密钥由本地桌面应用管理；不为了方便擅自上传敏感信息。

## 3. 目标用户与核心任务

### 3.1 主要用户

MVP 聚焦已经使用或愿意配置 API、每个项目需要多轮生成与修改的独立 AI 视觉创作者、创意技术人员和小型工作室主创。共同痛点是结果散落、上下文丢失、修改不可控；但他们不应该被迫在第一屏理解 Pipeline、Recipe、Lineage、Capability 等工程概念。

#### A. 有明确画面但不懂模型参数的创作者
- 能描述感觉、故事和参考作品，但不会写稳定 prompt；
- 需要快速把模糊想法变成可比较的画面；
- 默认使用步骤视图与简单修改。

#### B. 需要高控制度的设计师、导演和 AI 创作者
- 需要锁定角色、构图、镜头、风格或品牌元素；
- 会比较模型、参数、参考权重和迭代差异；
- 可以展开自由画布和高级参数，但不是首次使用的默认路径。

### 3.2 MVP 明确不服务

- 没有 API、只希望免费一键出图的普通消费者；
- 需要多人实时协作、SSO、审批审计的企业客户；
- 希望首版就构建任意 DAG 和插件执行环境的纯节点工具用户；
- 要求跨 Provider 像素级确定性、完整影视后期或专业数字资产管理的团队。

### 3.3 Jobs To Be Done

1. 当我脑中有一个不完整的视觉想法时，帮助我把它澄清为可执行的视觉方案。
2. 当第一次生成不准确时，让我只修改需要变化的部分，不破坏满意部分。
3. 当我在多个模型和供应商之间选择时，告诉我当前连接能做什么，而不是让我记 API 差异。
4. 当我同时探索多个方向时，让每个会话保持完整脉络，并能把好结果带到另一个分支。
5. 当我从图片继续做视频时，保留角色、场景、镜头和风格连续性。
6. 当我再次打开项目或交给其他人时，能够解释结果如何产生并尽可能复现。

## 4. 核心概念

### 4.0 用户术语与内部术语边界

P0 默认界面使用左列词；右列只允许出现在高级详情、开发文档、导出清单或调试信息中。

| 用户界面用词 | 内部/高级术语 | 默认展示规则 |
|---|---|---|
| 开始创作 | New Project / Pipeline | 首页只放一个主入口，不让用户先选模式 |
| 创作说明 | Creative Brief | 用户确认“理解是否正确”，不要求懂 brief |
| 步骤视图 / 创作步骤 | Pipeline Projection | 默认主工作区；不把 Pipeline 当成用户必须选择的产品模式 |
| 自由画布 | CanvasDocument | 高级展开入口，不与步骤视图同级抢首屏 |
| 生成设置 | Generation Recipe | 默认自动保存；详情中可查看/导出 |
| 来源记录 | LineageGraph | 选中作品后查看，不常驻主屏 |
| 本轮参考 | ContextManifest / ContextSnapshot | 默认一行摘要，展开后看来源与排除原因 |
| 连接 | Provider / CapabilityOffer | 先回答“能不能生成图/视频/对话”，技术证据折叠 |
| 助手 | AgentRun | 展示阶段、证据和停止，不展示内部思维链或角色群聊 |
| 另开方向 | Fork / Branch | 用创作语言表达分支，不用 Git/工程隐喻 |
| 版本 | ArtifactRevision / Attempt | 用户看到作品版本，工程记录保留 Attempt |

### 4.1 Project（项目）

最高层创作容器，包含：
- 项目简报与目标；
- 共享记忆与固定素材；
- 多个会话；
- 步骤视图与自由画布；
- 生成任务与来源记录；
- 连接引用（不在项目文件中保存明文密钥）。

### 4.2 Conversation（会话）

项目内的独立探索分支。每个会话拥有自己的原始消息、上下文快照、任务和素材引用。会话之间默认隔离，可显式：
- 另开方向：从当前上下文创建分支；
- Pin：把结论或素材提升到项目记忆；
- 引用另一条记录：引用另一会话的指定快照；
- 合并创作说明：只合并确认后的创意简报，不盲目拼接全部聊天记录。

### 4.3 Creative Brief（创意简报）

用户意图的结构化中间层，不等同于最终 prompt。最小字段：
- 目标与用途；
- 主体；
- 场景与环境；
- 构图/镜头；
- 动作与时间；
- 风格与材质；
- 光线与色彩；
- 必须保留；
- 必须避免；
- 参考素材；
- 不确定项与待确认问题。

### 4.4 Generation Recipe（生成配方）

一次可执行生成任务的冻结快照：
- brief 版本；
- prompt / negative prompt；
- 模型、连接和能力版本；
- 比例、分辨率、时长、seed（若支持）；
- 参考素材及其 hash/权重；
- 蒙版/区域约束；
- Provider 特有参数；
- 父素材与修改说明；
- 创建者（用户/助手，内部记录 AgentRun）和时间。

### 4.5 来源记录（内部 Lineage）

输出与其父版本、来源素材、生成设置、会话和任务之间的有向关系。来源记录必须支持：
- 查看父子版本；
- 对比 prompt/参数差异；
- 回到任一版本继续分支；
- 标记“最终”“候选”“废弃”；
- 导出生成设置和素材清单。

## 5. 产品信息架构

Simple First 后，产品信息架构按“用户任务”组织，而不是按底层模型组织。

```text
Gravuresse
├── 首页
│   ├── 开始创作（唯一主入口）
│   ├── 最近项目
│   └── 高级入口：导入项目 / 自由画布 / 连接设置
├── 创作工作区
│   ├── 创作记录（可折叠）
│   ├── 步骤视图（默认主工作区）
│   │   ├── 描述想法
│   │   ├── 确认创作说明
│   │   ├── 生成候选
│   │   ├── 选择结果
│   │   └── 简单修改
│   ├── 输入栏（描述 / 修改 / 参考素材）
│   ├── 详情检查器（默认隐藏，选中作品后打开）
│   └── 高级：自由画布 / 来源记录 / 生成设置
└── 设置
    ├── 添加连接向导
    ├── 已连接能力：可对话 / 可生成图 / 可生成视频
    └── 高级技术详情：endpoint / 模型 / 验证证据 / 请求模板
```

默认用户路径不得出现四等分顶栏“项目 / Pipeline / 无限画布 / 连接”。这些能力应按使用时机出现：

- 首次启动：只问“你想做什么”；
- 生成前：只提示缺少的连接；
- 选中作品后：才显示来源、版本、生成设置；
- 用户主动点击高级：才进入自由画布或技术详情。

## 6. 核心功能需求

优先级：P0 = Simple First 基础闭环；P1 = 稳定后扩展；P2 = 未来探索。

### 6.1 项目与默认入口

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| PROJ-01 | 首页只有一个主入口“开始创作” | P0 | 新用户不需要先理解 Pipeline / 自由画布；30 秒内可开始输入想法 |
| PROJ-02 | 新建后默认进入“步骤视图” | P0 | 步骤视图使用同一 Project 和 WorkflowGraph，不创建第二种文件格式 |
| PROJ-03 | 自由画布作为高级入口出现 | P0 | 文案为“展开为自由画布”，不与开始创作同级抢入口 |
| PROJ-04 | 选中作品后才打开详情检查器 | P0 | 未选中时不常驻参数面板，避免认知压力 |
| PROJ-05 | 项目自动保存、版本迁移、导入导出 | P0 | 崩溃恢复后保留最近一次完整写入 |

### 6.2 步骤式创作主路径

默认主路径只包含用户能理解的最小创作链：

```text
描述想法
→ 确认创作说明
→ 生成候选
→ 选择结果
→ 简单修改
→ 导出 / 继续做视频
```

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| STEP-01 | 每一步只问一个问题 | P0 | 新建问“你想做什么”；生成问“哪个更接近”；修改问“保持什么、改变什么” |
| STEP-02 | 步骤卡显示输入、结果、状态和下一步 | P0 | 用户知道当前卡在哪里，但不需要理解 WorkflowGraph |
| STEP-03 | 支持跳过可选步骤和从任一步重跑 | P0 | 不强制完整线性流程 |
| STEP-04 | 候选结果支持人工选定 | P0 | 未选定前不自动覆盖下游 |
| STEP-05 | 自动选择连接时只展示人话理由 | P0 | 例如“当前连接支持图生视频”，技术证据折叠到详情 |
| STEP-06 | 自定义步骤模板 | P1 | 本版只保留系统默认主路径 |

### 6.3 对话、上下文与创作记录

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| CTX-01 | 同一项目支持多个创作记录/会话 | P0 | 切换记录不串消息、不串素材状态 |
| CTX-02 | 会话可“另开方向” | P0 | 内部保留 Fork 来源，UI 不默认使用工程词 |
| CTX-03 | 项目记忆与会话历史分层 | P0 | 只有用户 Pin 的内容自动进入项目共享层 |
| CTX-04 | 输入前显示一行“本轮参考”摘要 | P0 | 例如“创作说明 · 当前记录 · 2 张参考图”；展开后看来源与排除原因 |
| CTX-05 | 超出上下文预算时生成版本化摘要 | P0 | 原文不删除；摘要可检查、回滚、重建 |
| CTX-06 | 独立窗口与多窗口同步 | P1 | 有 revision/冲突规则后开放 |
| CTX-07 | 详细上下文预算和裁剪原因 | P1 | 默认隐藏，避免把用户带入调试心智 |

### 6.4 助手辅助与 P1 协作

默认不是角色扮演群聊，也不是可见多 Agent 集群，而是一个顺序执行的“助手正在处理”状态：

```text
理解画面
→ 准备两个方向
→ 检查人物和构图是否一致
→ 选择可用连接
→ 等用户确认
```

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| AST-01 | 单协调助手执行创作说明、候选方向和连续性检查 | P0 | 首发不依赖可见多 Agent 集群；步骤可测试、可停止 |
| AST-02 | 运行前显示人话步骤和停止按钮 | P0 | 默认不显示模型、调用次数和内部任务图；详情可展开 |
| AST-03 | 助手输出结构化结果，不把内部闲聊塞进主会话 | P0 | 主会话只保留结论、差异和证据 |
| AST-04 | 候选方向支持差异说明 | P0 | 至少说明每个方向改变了什么，用于用户选择 |
| AST-05 | 连续性检查覆盖主体、场景、构图、光线、材质和禁用项 | P0 | 输出可直接更新创作说明或生成设置 |
| AST-06 | 受控多 Agent 协作和自定义 Agent 方案 | P1 | 只有任务持久化、审批、取消和恢复成熟后启用 |
| AST-07 | 自动多轮优化有次数/成本/时间上限 | P1 | 绝不无限自主循环 |

### 6.5 简单修改与高级设置

默认不是“黑盒/灰盒/白盒三标签”，而是一个简单修改面板：

```text
保持：人物、红伞、构图
改变：时间改清晨、雨改薄雾、灯光变自然
[生成修改版] [高级设置]
```

高级设置展开后才显示稳定锁、参考权重、Prompt diff、seed、模型、参数和来源记录。

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| EDIT-01 | 默认修改只显示“保持/改变” | P0 | 用户无需理解黑盒、灰盒、白盒即可完成修改 |
| EDIT-02 | 高级设置可展开稳定锁、变化强度、参考权重 | P0 | 展开前不占据主路径注意力 |
| EDIT-03 | 技术详情可查看 Prompt/参数差异、模型和 seed | P0 | 信息来自真实能力；不支持字段明确标记不可用 |
| EDIT-04 | 修改采用非破坏分支 | P0 | 原素材和原生成设置保持不变 |
| EDIT-05 | 来源记录与可重做程度默认收起 | P0 | 选中作品后可查看；不在首屏用 MEDIUM/LOW 刺激用户 |
| EDIT-06 | 区域修改、蒙版与局部重绘 | P1 | 仅在模型能力支持时出现 |
| EDIT-07 | 角色/风格一致性包 | P1 | 可将多素材与约束保存为可复用资产 |

### 6.6 图像与视频生成

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| GEN-01 | 文生图、图生图/编辑 | P0 | 根据连接能力显式启用 |
| GEN-02 | 图生视频作为可选下一步 | P0 | 不在首次生成前要求用户理解视频任务 |
| GEN-03 | 每次生成先冻结生成设置 | P0 | 输出失败也保留请求快照和错误 |
| GEN-04 | 支持候选批次、进度、取消、重试 | P0 | 重试不覆盖原任务记录 |
| GEN-05 | 图片可作为视频关键参考 | P0 | 来源关系写入来源记录 |
| GEN-06 | 视频延长、首尾帧、多镜头序列 | P1 | 按能力逐步开放 |
| GEN-07 | 成本估计与预算上限 | P1 | 无定价信息时明确显示“未知” |
| GEN-08 | 应用重启后恢复异步任务 | P0 | 已取得远程任务 ID 的继续轮询；状态未知的任务不自动重提，避免重复计费 |

### 6.7 连接设置

连接页默认是向导，不是开发者控制台。

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| CONN-01 | 提供“添加图像生成 / 视频生成 / 对话连接”向导 | P0 | 用户先按任务选择，而不是先理解 Provider 类型 |
| CONN-02 | 已连接列表用“可对话 / 可生成图 / 可生成视频”展示能力 | P0 | 不把 endpoint、revision、Bearer 等技术字段放在默认列表 |
| CONN-03 | 支持 OpenAI-compatible 与通用自定义请求模板 | P0 | 高级请求模板默认折叠 |
| CONN-04 | 拉取模型、测试连接、最小真实推理验证 | P0 | “能列模型”和“能生成”分开，但用人话显示 |
| CONN-05 | 密钥仅由主进程处理，渲染进程只见脱敏状态 | P0 | 日志、导出和项目文件无明文密钥 |
| CONN-06 | 连接可按项目设默认，也可单任务覆盖 | P0 | 切换有明确作用域 |
| CONN-07 | Provider 特有参数进入 namespaced extraParams | P1 | 不污染通用生成设置字段 |
| CONN-08 | 不在本版新增未经确认的预设 Provider | P0 | 保留兼容性；新能力优先通过自定义连接接入 |

### 6.8 自由画布与输出管理

自由画布是高级工作区，不是新用户必须理解的第二产品模式。

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| CAN-01 | 步骤视图可展开为自由画布 | P0 | 保留节点状态、素材、任务和连接关系；入口标记为高级 |
| CAN-02 | 图像、视频、文本、创作说明、任务引用可放入画布 | P0 | 节点有稳定 ID 和类型 |
| CAN-03 | 来源关系默认只显示选中作品的一跳 | P0 | 避免画布持续杂乱 |
| CAN-04 | 从任一素材发起修改、变体或视频任务 | P0 | 自动创建下游节点和来源记录 |
| CAN-05 | 支持分组、框选、缩放、平移、撤销/重做 | P0 | 保留现有能力并统一状态模型 |
| CAN-06 | 自由画布压回步骤视图时做可表示性检查 | P0 | 不可表示内容只读保留，不静默丢失 |
| CAN-07 | 时间线/镜头序列 | P2 | 不进入当前基础版本 |

## 7. 关键用户流程

### 7.1 从想法到第一组图

```text
打开首页 → 点击“开始创作”
→ 用自然语言描述想法 + 添加参考
→ 助手补齐缺失信息
→ 用户确认“创作说明”
→ 系统生成 2–4 个候选方向
→ 用户选择最接近的一张
→ 系统自动保存生成设置和来源记录
```

### 7.2 简单修改

```text
选择候选 → 说“保留人物和构图，把时间改成清晨”
→ 系统显示：保持人物/红伞/构图，改变时间/光线/天气
→ 用户点击“生成修改版”
→ 创建非破坏版本 → 生成
→ 用户可在详情中查看来源记录和生成设置差异
```

### 7.3 从图到视频

```text
选定图片 → 点击“继续做视频”
→ 助手补齐镜头、动作、节奏、时长
→ 用户确认视频创作说明
→ 系统选择支持图生视频的连接
→ 排队 → 轮询 → 生成视频版本
```

### 7.4 多方向探索与合并

```text
记录A探索写实方向 → 另开方向B探索插画方向
→ 两边历史互不污染
→ 将A的角色设定和B的色彩方案 Pin 到项目
→ 新记录显式引用两份项目记忆
```

### 7.5 展开为自由画布

```text
步骤视图中点击“高级：展开为自由画布”
→ 每个步骤变成节点组
→ 候选输出保留在对应步骤下
→ 当前选中路径高亮
→ 用户可自由整理素材、参考图和分支
```

## 8. MVP、P1 与非目标

### 8.1 vNext MVP（必须形成闭环）

1. 首页单主入口“开始创作”与默认步骤视图；
2. Project 新模型、自动保存与主进程权威写入；
3. 创作说明、候选生成、选择结果、简单修改的主路径闭环；
4. 单协调助手 + 候选方向 + 连续性检查（顺序执行，默认人话状态）；
5. 多创作记录隔离、“另开方向”、Pin、本轮参考摘要；
6. 简单修改默认面板，高级设置折叠稳定锁、参数差异和来源记录；
7. 步骤视图可高级展开为自由画布，但不作为新用户首屏同级模式；
8. 图像/视频统一任务队列与重启恢复；
9. 向导式连接设置与能力驱动的自定义 API 连接；
10. 来源记录、可重做说明、错误与迁移机制。

**Go/No-Go：** 如果首发仍要求用户先理解“Pipeline / 无限画布 / 黑灰白盒 / Provider / Capability / Recipe / Lineage”等内部概念，或重新把可见多 Agent 集群、任意工作流 DAG、全 Provider 全参数拉回 P0，则判定 No-Go。先完成“描述 → 生成 → 选择 → 修改 → 导出/恢复”的简单闭环。

### 8.2 P1

- 自定义步骤模板 / Pipeline 模板；
- 受控多 Agent 集群与自定义 Agent 方案；
- 区域编辑、角色/风格一致性包；
- 独立会话窗口；
- 成本预算和使用统计；
- 视频延长、首尾帧和更强镜头编排；
- 专家模式下更完整的白盒参数工作台。

### 8.3 P2 / 暂不做

- 完整专业非编系统；
- 多人实时协作；
- 云端账号、团队权限、素材市场；
- 模型训练/LoRA训练平台；
- 自动发布到第三方平台；
- 插件市场；
- 全 Provider、全模型、全参数无差别兼容；
- 无限制自主 Agent 循环。

## 9. 成功指标

### 9.1 核心指标

- **30 秒开始率**：新用户在 30 秒内从首页进入输入想法状态的比例；
- **3 分钟首轮闭环率**：新用户在 3 分钟内完成“描述 → 生成 → 选择 → 修改”的比例；
- **意图收敛轮次**：从初始描述到用户确认创作说明所需轮数；
- **满意结果命中率**：一次候选批次中至少出现一个被用户收藏/选定结果的比例；
- **局部修改保留率**：用户声明的保持项在下一轮仍被主观判定为保留的比例；
- **来源可追溯率**：输出能够还原有效生成设置和父素材的比例；
- **上下文误引用率**：模型错误引用其他创作记录私有信息的比例；
- **任务成功率**：经过能力预检后实际完成生成的任务比例；
- **任务恢复正确率**：故障注入后不丢失已完成资产、不重复提交可能计费任务的比例；
- **助手增益**：相同创作说明下，助手辅助候选相对“原始提示词直出”的盲测胜出/持平率，并同时记录额外时间与成本。

### 9.2 体验指标

- 新用户能在不理解 Pipeline、Recipe、Lineage、Provider 等术语的情况下完成“想法→图片→修改”；
- 首页只有一个主入口，用户不需要先选择工作流模式；
- 专业用户能在详情中查看并修改完整生成设置；
- 用户能在 10 秒内理解当前任务正在做什么、下一步是什么、能否停止；
- 任何高成本或批量调用均可在执行前确认、执行中停止。

## 10. 验收场景

### AC-01：Simple First 生图闭环

给定一个新用户输入“雨夜里的中式未来城市，一位撑红伞的人走过老街”，用户从首页点击一次“开始创作”即可输入想法；系统生成创作说明并得到候选图。过程中不得要求用户先选择 Pipeline/自由画布、填写 seed/采样参数或理解 Provider。

### AC-02：高级详情不打扰主路径

选择任一输出，默认检查器只显示“版本、来源、可继续做什么”。用户点击“查看生成设置/技术详情”后，才能看到可获得的连接、模型版本、prompt、negative prompt、比例、分辨率、seed、参考素材、父版本与 Provider 参数；不支持字段明确标记不可用，而不是伪造默认值。

### AC-03：简单修改

用户要求“只把雨夜改成清晨，人物、红伞和构图不变”，系统先以人话显示“保持/改变”摘要，再把锁定项和变化项写入新生成设置，生成结果作为原图子版本保留。

### AC-04：多创作记录隔离

项目有 A/B 两个创作记录；B 不引用 A 时，不应自动注入 A 的私有消息。A 的内容只有 Pin 到项目或被显式引用后才可用于 B。

### AC-05：步骤视图 / 自由画布转换

含创作说明、两组图像候选和一项视频任务的步骤视图能通过“高级：展开为自由画布”展开为画布，节点、状态、候选和来源记录完整；切回步骤视图时，无法表达的自由节点被保留并给出提示。

### AC-06：连接向导与安全

用户能通过“添加图像生成连接 / 添加视频生成连接 / 添加对话连接”向导完成配置。默认列表只显示可用能力和脱敏状态；导出项目、应用日志、渲染进程状态和错误信息中不得出现明文 API Key、token、Cookie 或认证头。

### AC-07：异步任务恢复

图生视频任务提交并取得远程任务 ID 后强制退出应用；重启后系统继续轮询并回写同一 TaskRun。若崩溃发生在无法确认是否提交成功的同步调用阶段，任务进入 `unknown`，不得自动重提。

## 11. 主要风险与对策

| 风险 | 后果 | 对策 |
|---|---|---|
| 复杂度过早暴露，用户第一眼被吓退 | 产品被误认为工程工作站，小白不敢开始 | 首页单主入口；默认步骤视图；高级概念折叠；UI 术语表门禁 |
| 目标过大，一次重写所有功能 | 长期无法发布 | 复用现有 Provider、会话、画布与任务队列；按领域模型增量迁移 |
| Agent 协作成本高但提升有限 | 用户付费增加、等待变长 | 默认单助手；复杂任务才出现协作建议；显示调用预算 |
| 不同 Provider 参数差异巨大 | 技术详情界面失控 | 通用生成设置 + capability schema + namespaced extraParams；默认界面只显示“能做什么” |
| 上下文摘要漂移 | 创意设定被错误改写 | 摘要版本化、保留原文、用户可 Pin/纠正/回滚 |
| “稳定锁”被用户理解为绝对保证 | 信任受损 | 默认用“尽量保持”；详情解释可重做程度和模型能力边界，不使用“100%一致”措辞 |
| 步骤视图与自由画布双向同步复杂 | 数据丢失或状态冲突 | 共享图模型；步骤视图是受限投影，不维护第二份业务数据 |
| API 密钥和自定义端点引入安全风险 | 凭据泄漏/SSRF | 主进程持有密钥、端点验证、IPC allowlist、日志脱敏、导出过滤 |
| 多窗口并发写入 | 会话覆盖 | P1 再开放；引入 revision/optimistic concurrency 后实施 |
| 视频/远程任务在退出后丢失或重复计费 | 资产丢失、成本失控 | 主进程持久化 TaskRun/Attempt；先记录远程任务 ID，再轮询；未知状态人工确认 |

## 12. 变更规则

后续想法按以下格式进入 PRD，不直接塞进开发任务：

```text
新想法：
解决谁的什么问题：
进入哪个核心流程：
是否改变数据模型：
优先级建议：P0 / P1 / P2
不做会造成什么：
```

只有经过产品范围审查的需求，才能进入 `SPEC.md` 和开发任务。
