## 一、产品定位与核心命题

### 1.1 一句话定义

**Gravuresse 是一个"对话驱动的 AI 创意设计桌面工具"**——用户通过自然对话表达创意想法，AI 理解意图后自动转化为精准的生成提示词，调度图像/视频模型产出内容，并在无限画布上组织和管理这些产出。

### 1.2 为什么存在

现状中存在三组矛盾：

| 矛盾 | 描述 |
|------|------|
| **模型多≠好用** | Midjourney、DALL·E、Gemini、Seedream……每家模型各有所长，但用户被锁在各自的 UI 和交互范式里，切换成本高 |
| **提示词门槛** | 设计专业学生/非技术用户的表达方式是"自然语言描述想法"，而不是"写 prompt"。当前工具要求用户自己掌握 prompt engineering |
| **费用焦虑** | 好用的 AI 设计工具订阅费高（Midjourney $10-60/月 + ChatGPT $20 + 视频生成另算），学生难以长期承受 |
| **订阅碎片化** | 用户可能同时有 Midjourney 会员、Krea 会员、Runway 会员，但每次只能在一个工具里用——缺乏一个统一的入口来管理所有已购产能 |

> ⚠️ **坦诚说明：BYOK 并不一定比订阅便宜**
> DALL·E API 一张图约 $0.04-0.08，100 张 ≈ $4-8，加上 chat 开销，实际使用成本可能接近甚至超过 Midjourney 订阅。Gravuresse 的"费用优势"不在单价更低，而在**按需付费、不浪费**——你不必为没用到的月度额度付费，也可以根据预算精确控制生成次数。定价定位是"弹性"而非"便宜"。

Gravuresse 的核心假设：**AI 应该理解"你想要什么"，而不是等你写好 prompt 再执行**。

### 1.3 目标用户画像

| 属性 | 描述 |
|------|------|
| **身份** | 设计专业本科生/研究生，独立创作者 |
| **痛点** | 有想法但 prompt 写不好；在多个 AI 工具间切换心累；订阅费超预算 |
| **技术能力** | 会使用 Figma/PS 等设计工具，但不愿学 ComfyUI 节点编辑或写复杂 prompt |
| **工作流** | 先粗想→和 AI 对话 refine→生成素材→放入画布整理→反复迭代 | |
| **订阅资产** | 可能已购 Midjourney 会员、Krea 会员、Runway 会员等，希望在一个界面里统一管理所有 AI 产能 | |
| **非目标** | 专业 AI 画师（用 Stable Diffusion + ControlNet）、平面设计师（用 PS/AI）、3D 设计师 | |

### 1.4 使用场景优先级

| 场景 | 优先级 | 频率 |
|------|--------|------|
| 🟢 「我有一个想法，帮我生成一张图」→ 对话 → 出图 | P0 🎯 | 高频 |
| 🟢 不满意结果，自然语言描述修改 → AI 增量调整 prompt → 重生成 | P0 🎯 | 高频 |
| 🟡 生成的图片在画布上整理、对比、挑选 | P1 | 中频 |
| 🟡 将本地图片导入画布作为素材/参考 | P1 | 中频 |
| 🟡 同时管理多个项目的素材和对话 | P2 | 中频 |
| 🟡 生成短视频素材（产品展示、动态视觉） | P2 | 低频 |
| 🔵 在画布上手绘草图辅助表达 | P3 | 低频 |
| 🔵 批量生成多张相同 prompt 的变体 | P3 | 低频 |

### 2.2.1 模块对标基准

Gravuresse 由三大核心模块组成，每个模块有明确的对标产品：

| 模块 | 对标产品 | 对标维度 |
|------|---------|---------|
| **自由画布** | [Lovart](https://lovart.dev) | 无限画布的交互质感：缩放流畅度、资产排列自由性、视觉反馈（shimmer 动效等） |
| **视频生成** | [libtv](https://libtv.app) | 视频生成的任务编排、进度追踪、多模型切换、结果管理 |
| **对话生成** | — | 自研对话驱动范式，无直接对标 |

> Gravuresse 不是想做 Lovart 或 libtv 的替代品，而是把它们的**最佳体验**整合到一个统一的"对话驱动"工作流里。
>
> 🟡 **提醒：** Lovart 和 libtv 都是垂直领域的深度产品。Gravuresse 在整合时要在"深度"和"广度"之间做取舍——初期不做每个模块的全部功能，而是抓住每个模块最核心的 20% 体验。

### 2.2.5 为什么是桌面端？（架构决策溯源）

> 🟡 当前没有做严格的 A/B 验证，以下是选择 Electron 的理由和后验评估：
>
> **当初选择桌面端的理由：**
> - 本地文件直接管理（把生成的图片存到本地文件夹，不用上传/下载）
> - 离线 UI 操作（编辑对话、排列画布不需要网络）
> - 窗口管理（多显示器、分屏工作流）
> - 性能（Web 版无限画布在大素材量时容易掉帧）
>
> **事后看，这个选择是否合理？**
> - ✅ 文件管理确实是优势——用户生成的素材自动进本地文件夹
> - ⚠️ 但本地部署也带来了安装/更新的摩擦（目前没有自动更新机制）
> - ⚠️ Web 版理论上可以覆盖更多用户（尤其是 Mac 用户被排除在外）
>
> **结论：** 继续桌面端路线，但需要补自动更新。未来如果有 Mac 需求，Electron 可以跨平台。

### 2.3 Gravuresse 的独特优势

1. **对话即界面** — 不是"写 prompt → 出图"，而是"描述想法 → AI 帮你优化 → 确认 → 生成"。这个 flow 更接近设计创作的自然过程
2. **统一入口，管理所有 AI 产能** — 一个客户端管理多家 API Key 和会员订阅，用户只需记住一个交互范式
3. **模块化架构** — 生图、视频、画布是独立模块，各自有专门的交互空间，互不干扰，可独立升级
4. **桌面端离线可用** — 界面操作不依赖网络，只有 API 调用时在线，比 Web 应用响应更快
5. **BYOK + 订阅接入** — 支持自带 API Key，也支持绑定已有会员账号（Midjourney 订阅、Krea 会员等），用户已有的产能不浪费

### 2.4 差异化叙事

> 其他 AI 设计工具让你**写 prompt**。Gravuresse 让你**聊天**。

### 2.5 真正的护城河是什么？（自我拷问）

> 🟡 **坦率说：当前版本的 Gravuresse 没有护城河。**
>
> "对话驱动"的差异化在功能层面很容易被复制——Midjourney、Krea 随时可以加一个自然语言 prompt 优化层。Gravuresse 真正的长期价值不在 prompt 优化，而在：
>
> 1. **上下文持久性** — AI 了解你的整个项目历史，不是一次性对话（当前部分实现）
> 2. **跨模型一致性** — 同一套交互范式管理多个生成服务（当前已实现）
> 3. **本地文件即资产** — 素材管理在本地，不依赖云端画廊（当前已实现）
>
> 这些是 Web 工具不容易做到的。要把护城河真正建起来，需要**让 AI 记住你的风格偏好、项目上下文、迭代历史**，这比"帮你写 prompt"难复制得多。当前的实现还没走到这一步。

## 三、产品哲学与设计原则

### 3.1 核心哲学：**对话优先，画布其次**

Gravuresse 不是"一个带聊天功能的画布工具"，而是"一个以对话为驱动、以画布为整理空间的创意伴侣"。

- **对话是创造引擎** — 所有的生成、修改、迭代都在对话中进行
- **画布是整理空间** — 生成的素材自动流入画布，供你排列、对比、挑选
- **设置是幕后控制** — 模型、参数、API 配置在后台，不打断创作流

### 3.2 设计原则

| 原则 | 含义 | 设计决策示例 |
|------|------|-------------|
| **先审后出** | 任何生成前先展示 AI 理解的 prompt，用户确认后再调 API | 当前 v1.1 已实现 |
| **修改不说重做** | 迭代修改基于上次 prompt 增量调整，保留满意部分 | 当前已实现 |
| **一应用三轨道** | Chat / Image / Video 三条轨道各自独立配置，但共用同一个交互空间 | 当前已实现 |
| **配置不干扰创作** | API Key、模型选择等设置在 Settings 面板，对话和画布不受影响 | 当前已实现 |
| **结果可追溯** | 每张素材都知道"从哪个对话、哪个 prompt、哪个模型来的" | 当前部分实现（asset 记录了 prompt+model，但未建立到 message 的引用） |

### 3.3 视觉设计方向

> 🟡 **坦诚评估：当前（v1.5）的界面设计太平庸。**
>
> 功能撑起来了，但视觉停留在"能用"水平——默认主题缺乏性格、组件间距和色彩层级没有体系化设计、缺少让人"wow"的细节。而 Lovart、Krea 这些对标产品的第一印象恰恰是视觉质感。

**定位：** Gravuresse 的视觉应该是"安静的精密感"——不喧哗但有细节，不花哨但有品质。

**方向性要求：**

| 维度 | 目标状态 | 反例 |
|------|---------|------|
| **色彩体系** | 低饱和度主色调 + 金色/琥珀色作为点缀色（呼应创意/灵感氛围） | 纯白/纯黑默认主题 |
| **字体排版** | 清晰的信息层级，对话和画布内容为主的布局，不过度设计"壳" | 边框过重、装饰元素喧宾夺主 |
| **动效** | 克制的微动效：生成 shimmer、面板切换、hover 反馈——不打扰但让交互有"质感" | 无动效或过度动画 |
| **画布质感** | 对标 Lovart 的画布底色、阴影、选中态反馈、缩放手感 | 纯白画布+硬边矩形 |
| **暗色模式** | 真正的暗色设计（不仅仅是反色），深灰底+柔和发光 | 纯黑底+刺眼白字 |

**实施路径（不在 PRD 阶段深究，留到 Spec/设计稿阶段）：**
1. 定义 Design Token（色彩、间距、圆角、阴影、字体）——先于 UI 实现
2. 重新设计主界面布局（左侧导航+独立模块面板）
3. 画布视觉升级（底纹、网格、阴影、选中态）
4. 动效库（渐入、 shimmer、过渡）

## 四、功能架构与数据流

### 4.1 系统模块图（模块化架构）

Gravuresse 由三个对等的独立模块组成，**各自有独立的 UI 面板**，通过对话模块串联：

```
┌──────────────────────────────────────────────────────────┐
│                       TitleBar                           │
│             窗口控制 | 设置 | 版本号                      │
├─────────────────────────┬────────────────────────────────┤
│   左侧导航               │         主工作区                │
│   ┌─────────────────┐   │  ┌──────────────────────────┐  │
│   │ 💬 对话          │   │  │ (当前激活的模块面板)       │  │
│   │ 🖼️ 生图          │   │  │                          │  │
│   │ 🎬 视频          │   │  │  对话 → ChatPanel        │  │
│   │ 🎨 画布          │   │  │  生图 → ImagePanel       │  │
│   └─────────────────┘   │  │  视频 → VideoPanel       │  │
│                         │  │  画布 → CanvasPanel      │  │
│   导航切换，模块独立     │  │                          │  │
│   当前模块占据主工作区    │  └──────────────────────────┘  │
├─────────────────────────┴────────────────────────────────┤
│                      ModelBar                            │
│   Chat模型 | Image模型 | Video模型 | 全局任务进度          │
└──────────────────────────────────────────────────────────┘
```

> 🏗️ **当前状态 VS 目标状态**
>
> | 方面 | 当前 (v1.5) | 目标 (v2.0) |
> |------|-------------|-------------|
> | UI 布局 | 对话+画布平铺在中间主区 | 左侧导航，模块独立切换 |
> | 模块耦合 | 生图/视频嵌入对话流，无独立面板 | 各自有独立交互面板，对话作为串联层 |
> | 画布 | 与对话在同一视图 | 独立全屏画布空间 |

> 🟡 **迁移策略：** 不需要一次性重构 UI 布局。可以先用"对话→生图→画布"的流转闭环验证模块化逻辑，UI 布局在 Phase C 迭代。

## 五、功能分级定义

### P0 — 核心闭环（对话出图 → 修改 → 使用）

| 功能 | 当前状态 | 说明 |
|------|---------|------|
| 对话驱动的图像生成 | ✅ v1.1 | AI 先出 prompt，用户确认后生成 |
| 自然语言迭代修改 | ✅ v1.1 | 基于上次 prompt 增量调整 |
| 多对话管理 | ✅ v1.1 | 独立消息+资产，切换不丢失 |
| 任务卡片实时状态 | ✅ v1.1 | 待确认→生成中→已完成/失败 |
| 图片缩放预览 | ✅ v1.2 | 滚轮缩放、拖拽平移、双击重置 |

**缺陷（待修复）：**
- ⚠️ 对话和画布资产的关联需要更清晰——当前 asset 记录 prompt+model，但没有指向源 message 的 ID，无法回答"这张图是怎么生成的"
- ⚠️ 提示词修改后缺少"查看修改历史"的能力

### P1 — 画布体验（整理、对比、构建）

> 🟡 **自省：画布真的是 P1 吗？**
>
> 无限画布（CSS transform 缩放、自由拖拽、绘图工具）是 Gravuresse 在 v1.2 投入了大量开发资源的功能。但坦白说，对于核心场景"对话出图→挑选→使用"而言，**一个结构化的画廊视图（网格排列+选中预览）可能已经覆盖 80% 的需求**。画布的缩放平移和自由放置功能，实际使用率可能远低于预期。
>
> 如果资源有限，画布可以降级为 P2——保持网格模式+基本展示，把研发精力投入到更核心的对话体验和 Provider 架构上。这个决定需要由数据支撑（如果已有用户使用数据的话），或者由你凭直觉判断。

| 功能 | 当前状态 | 说明 |
|------|---------|------|
| 无限画布（CSS transform） | ✅ v1.2 | 缩放平移，浮动控件 |
| 网格/自由模式切换 | ✅ v1.3 | 结构化 vs 自由排列 |
| 生成动效（shimmer 占位） | ✅ v1.3.1 | Lovart 风格 |
| 右键菜单（查看/下载/删除/重生成） | ✅ v1.3.1 | |
| 素材保存到本地 | ✅ v1.3 | `Pictures/Gravuresse/` |
| 导入本地图片到画布 | ❌ P1 待实现 | 支持拖拽和文件选择器导入，导入后成为画布资产，可用于参考/对比/作为 AI 生成基础 |

**缺陷/缺口：**
- ⚠️ 画布上没有搜索/筛选功能（对话多了，找某张图困难）
- ⚠️ 自由模式下没有"对齐到网格"的辅助线

### P2 — 多模型集成

| 功能 | 当前状态 | 说明 |
|------|---------|------|
| 三轨道独立 Provider | ✅ v1.1 | Chat/Image/Video 各自独立配置 |
| API Key 加密存储 | ✅ v1.5 | safeStorage |
| 自动获取模型列表 | ✅ v1.1 | 输入 API Key 后自动拉取 |

## 六、不做清单 (Anti-Features)

这些功能**不会**进入 Gravuresse，因为它们与产品定位相悖：

| 不会做 | 原因 |
|--------|------|
| **节点编辑器**（ComfyUI 风格） | 学习曲线高，违背"低门槛对话驱动"定位 |
| **社交/社区/分享广场** | 不是社交产品，功能膨胀 |
| **自建 AI 模型训练/微调** | 桌面端不适合做训练，交给专业平台 |
| **协作/多人实时编辑** | 桌面端单机工具，不是 Figma |
| **非设计类 AI 功能**（写作、翻译、代码） | 保持设计工具专注 |
| **Gravuresse 自建订阅系统**（用户付钱给 Gravuresse 买额度） | 不碰支付和额度管理，只做"产能整合"——用户已购的会员/BYOK 都在一个入口用 |

> 🟡 **关于"订阅接入"的解释：**
> Gravuresse 不做"订阅销售"，做"订阅统一管理"。用户已有的 Midjourney 会员、Krea 会员、Runway 会员等，Gravuresse 提供接入配置，**不代收代付**。这和 BYOK 是同一逻辑：用户已有的产能，Gravuresse 帮你统一调度。

## 七、技术架构原则

### 7.1 现架构评估

| 层级 | 现状 | 评估 |
|------|------|------|
| **UI 层** | React 18 + JSX | ✅ 够用，但组件颗粒度偏大 |
| **状态管理** | useState + useRef + props drilling | 🟡 适合当前规模，700+ 行的 CanvasPanel 是重构信号 |
| **IPC 层** | preload.js 定义 window.electronAPI | ✅ 安全隔离做得好 |
| **主进程** | main.js 288 行，IPC handler + API 调用 | 🟡 API 调用逻辑（chat/image/video）分散，没有 request pipeline |
| **存储层** | config.js (JSON) + store.js (JSON) | ✅ 原子写入+加密 Key |
| **Provider** | 三个常量 JS 文件（5-11 行） | 🔴 抽象太弱，添加新 provider 成本高 |

### 7.2 架构原则（模块化导向）

1. **不引入 TypeScript** — 当前 JSX 规模和团队规模（你一个人）不值得 TS 的编译配置和类型体操成本。用 JSDoc 标注关键接口即可。
2. **状态管理保持现状** — 当前 React state + ref 方案在 4600 行规模上是合理的。出现 1500+ 行 App.jsx 前不需要 zustand。
3. **模块化是第一原则** — 生图、视频、画布是独立模块，各自有：
   - 独立的 UI 面板（可以在左侧导航切换）
   - 独立的 Provider 配置（已实现）
   - **独立的数据流**（尽量不共用状态，通过对话模块做串联）
4. **Provider 层必须改造** — 这是目前最大的技术债。统一为一个 request pipeline + provider 注册表，使得添加新 API 服务只需要新增一个 provider 配置。
5. **CanvasPanel 需要拆但不要太碎** — 701 行拆成 3-4 个文件，而不是 10 个。
6. **订阅接入走 BYOK 相同的接口层面** — 不单独开发"订阅管理"UI，而是扩展 Provider 配置层，支持"API Key"和"会员 Cookie/Token"两种鉴权方式。
